System Requirements Specification (SRS)
Rider Payroll & Accounting ERP

1. SYSTEM OVERVIEW

This ERP manages riders, payroll, fines, bikes, expenses, journals, and cashflow for rider
outsourcing companies. Built on strict double-entry accounting principles (Journals)

Users:
1. PRO (limited access)
2. Accountant (Full/admin level access, Accountant is the admin)

STEP 1 -- CORE ACCOUNTING PHILOSOPHY (NON-NEGOTIABLE RULES)

This step defines rules that everything else depends on.
If these are wrong, the whole system collapses.

1.1 Journal Is the Absolute Root

Rule 1.1.1 -- Journal Mandatory

     NO transaction can exist without a Journal
             o Expense
             o Salary
             o Loan
             o Fine
             o Drawer balance correction
             o Capital injection
             o Profit withdrawal

     Even manual balance corrections MUST create a journal.

 Confirmed: YES

Rule 1.1.2 -- Journal Immutability

     Once a Journal is POSTED:
             o  Cannot be edited
             o  Cannot be deleted
             o  Can only be reversed
     Reversal creates:

             o A new journal
             o With equal and opposite ledger entries
             o Linked to original journal via reversal_of_journal_id

 Confirmed: YES (only reversed)

Rule 1.1.3 -- Double-Entry Enforcement

For every journal:

     Sum of Debit = Sum of Credit
     System hard-blocks save if imbalance exists

 No override, even for admin.

1.2 Ledger vs Expense (Final Terminology Lock)

Ledger (What it IS)

     Ledger is NOT a module for creating data
     Ledger is a read-only transactional history
     It answers:

             o Who owes how much?
             o Who was paid how much?
             o When did money move?

Ledger entries are generated automatically from journals.

Expense Module (What it IS)

     Expense module is a journal creation interface
     Accountant creates:

             o Expense journals
             o Loan journals
             o Salary journals
             o Fine journals
             o Manual adjustment journals
     Expense Categories:
             o Configurable
             o Drive default behavior, not accounting truth

 This matches your corrected understanding.

1.3 Receivable / Payable -- Mandatory Decision Point

Rule 1.3.1 -- Mandatory Flag

While creating ANY journal, system MUST require:

     Is this:
             o Receivable?
             o Payable?
             o Pure expense?

 Confirmed: Mandatory

Rule 1.3.2 -- What Receivable Means

If marked Receivable:

     Journal MUST include:
             o Entity Type (Rider / Vendor / Supplier)
             o Entity ID
             o Amount receivable

     This amount:
             o Appears in that entity's ledger
             o Appears in rider's payslip (if rider)

Example:

Visa cost = 2,000
Charged to rider = 5,000
 Receivable = 5,000 (NOT 2,000)

Rule 1.3.3 -- What Payable Means

If marked Payable:
     Creates liability in ledger
     Must be cleared later via payment journal

1.4 Drawers -- Absolute Money Containers

Rule 1.4.1 -- Drawer Is a Ledger Entity

     Drawer is NOT a balance field
     Drawer balance = sum of ledger entries affecting it
Drawers:
     Bank
     Cash
     Noqodi (Govt card)
     Others later

Rule 1.4.2 -- Manual Drawer Adjustments

Allowed via:
     Journal Type: Manual Adjustment
     Mandatory reason text
     Mandatory source drawer

Use cases:
     Capital injection
     Profit withdrawal
     Bank mismatch correction

STEP 2 -- RIDER IDENTITY, ALIASES & COMPANY SWITCHING (CRITICAL)

This is one of the most important parts of your system.
If this is wrong, payroll breaks silently.

2.1 Rider Identity -- Core Principle

Rule 2.1.1 -- Internal Rider ID (Primary Key)

     System generates a permanent internal rider ID
     This ID:

             o NEVER changes
             o Is NEVER visible externally
             o Is used for:

                       Ledger
                       Payroll
                       Fines
                       Expenses
                       Historical records

Example:

internal_rider_id = RDR-UUID-123

Rule 2.1.2 -- External IDs Are NOT Primary Keys

The following are NOT primary identifiers:

     Talabat ID
     Keeta ID
     Rider Name
     Bike Plate
     Phone number

They are aliases only.

Rule 2.1.3 -- Emirates ID (Long-Term Anchor)

     Emirates ID is:
             o Mandatory
             o Unique

     Used ONLY for:
             o Rare audits
             o Long-term record recovery

      Not used in daily searches
      Not used in payroll matching

This solves:

     Name duplication
     Rider ID changes
     Multi-year audits
2.2 Rider Aliases (This Solves 90% of Confusion)

Rule 2.2.1 -- Rider Alias Table

Each rider can have multiple aliases.
Each alias contains:

      internal_rider_id

     platform (Talabat / Keeta / Bykea / etc)
     external_rider_id (the ID used in salary sheets)

      valid_from_date

     valid_to_date (nullable)
     status (Active / Inactive)

Rule 2.2.2 -- One Active Alias Per Platform

For a given rider + platform:
     Only ONE alias can be active at a time

System MUST enforce this.

Payroll is NOT just salary.

3.1 Core Payroll Philosophy

Rule 3.1.1 -- Payroll Is System-Generated
     Payroll journals are ALWAYS system-generated
     Accountant cannot manually post salary journals
     Manual input = configuration only

 Already approved.

Rule 3.1.2 -- One Payslip Per Rider Per Month Per Platform
For a given:

     Rider (internal ID)
     Platform (Talabat / Keeta / etc)
     Month
 Exactly ONE final payslip can exist.
System MUST block duplicates.
3.2 Salary Sheet Ingestion (Parent Company)

Rule 3.2.1 -- Salary Sheet Upload

Accountant uploads:

     Platform-specific salary sheet
     Month + Year mandatory
     Platform mandatory

Sheet contains:

     External Rider ID
     Gross Salary
     Parent deductions (if any)
     Net paid by platform (optional)

Rule 3.2.2 -- Alias Resolution (Hard Gate)

For EACH row:
System tries:

    1. Match External Rider ID
    2. Match Platform
    3. Match Date within alias validity

If FAIL:

      Block payroll finalization
      Flag "Rider Alias Mismatch"
     Row stays unresolved

No bypass.

3.3 Pre-Payslip State (Staging Area)

Before finalization, system creates a Draft Payslip.

Draft Payslip contains:

     Gross salary (from platform)
     Platform deductions (read-only)
     Company deductions (editable)
     Outstanding balances snapshot

 No ledger entries yet.

Company deductions can come from:

     Loans / advances

     Any future receivable category

     Backed by ledger receivables

Rule 3.4.2 -- Manual Deduction Control

     Inputs:
             o How much to deduct THIS month
     System shows:
             o Remaining balance AFTER deduction (live)

This avoids tab switching -- as you requested.

Rule 3.4.3 -- Over-Deduction Block

System MUST block:
     Deduction > outstanding receivable
     Deduction > salary net available
No negative surprises.

      month
