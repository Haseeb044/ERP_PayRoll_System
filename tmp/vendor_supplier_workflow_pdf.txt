1. Purpose 
 
This document explains how Vendor and Supplier accounting should work within the ERP 
system. It covers all aspects of financial transaction management to ensure full transparency, 
accuracy, and traceability without blocking business operations. 
 
What This Document Covers 
• Profile creation flow for Vendors and Suppliers 
• VAT handling rules per party type 
• Journal posting logic and ledger impact 
• Payment scenarios: cash, credit, and partial payments 
• Adjustments, corrections, and audit trail behavior

4. UX Flow — Journal Screen 
 
Vendors and Suppliers can be created inline during journal posting — no need to navigate away 
from the journal form. 
 
Inline Profile Selection & Creation — Step by Step 
• 1. User selects party_type (Vendor or Supplier) 
• 2. User selects existing profile from searchable dropdown 
• 3. If profile does not exist → click 'Create New Vendor' or 'Create New Supplier' 
• 4. Mini-form opens on the same screen (modal flow) 
• 5. On save → profile inserted into master table 
• 6. Dropdown refreshes and auto-selects newly created profile 
• 7. User completes and posts the journal entry 
 
Validation Rules 
• Party type chosen → profile must be selected before posting 
• Vendor selected + VAT enabled → VAT calculation must be validated 
• Supplier flow → VAT lines are blocked entirely 
• Duplicate name → warning shown (not a hard block unless policy requires uniqueness) 
• Unbalanced journal → always hard-blocked

6. Journal vs. Ledger 
 
 
🏭  Journal 🏭  Ledger 
• The accounting instruction set for a 
transaction 
• The posted financial history generated from 
journals 
• Defines: debit expense, debit VAT input, 
credit payable 
• Used for: balances, statements, aging, 
audit, reporting 
• Must always balance: total debit = total 
credit 
• Old entries are never rewritten — 
corrections use new entries 
• Represents what was posted • Represents cumulative financial truth

10. Partial Payments 
 
Every payment creates a new journal and ledger entry. Previous payment rows are never 
overwritten. 
 
Scenario: Supplier Payable = 2,000 AED 
• First payment posted: 800 AED 
• Remaining outstanding: 1,200 AED 
• ERP shows: paid amount, remaining balance, due status 
 
Payment 1 of 2 — 800 AED

11. Adjustments, Corrections & Reversals 
 
Core Rule 
• Never silently edit posted financial history — all corrections must create new visible entries 
 
Correction Patterns 
Option A — Adjustment Entry Option B — Reversal + Repost 
• Post only the difference amount • Fully reverse original entry 
• Minimal impact to existing entries • Repost the correct entry 
• Used for small over/under posts • Used when entire transaction is wrong 
 
Example: VAT Correction (posted 60, should be 50) 
# Entry Type Account Amount (AED) 
1 Credit VAT Input Account 10.00 
2 Debit Correction Account 10.00 
→ NOTE Original entry remains visible Audit safe

13. Data Integrity & Safety Rules 
 
 
🏭  Hard Blocks (Always Enforced) ⚠  Soft Warnings (Show, Don't Block) 
• Journal must balance (debit = credit) • Negative net amount detected 
• Vendor VAT logic only in Vendor category • Missing VAT number for VAT vendor 
• Supplier flow produces no VAT lines • Unusually high VAT percentage 
• Posted entries protected from silent edits • Missing counterparty profile

17. Final Confirmation Summary 
 
 
01 Vendor and Supplier are handled as separate entity types with separate profile 
tables 
02 VAT (5%) applies only to the Vendor flow — never applied to Supplier transactions 
03 Every posting generates balanced journal entries (total debit = total credit) 
04 Ledger reflects all posted transactions providing complete financial transparency 
05 Payments (full or partial) correctly reduce the outstanding payable balance 
06 All adjustments and corrections are audit-safe — no hidden history edits allowed 
07 Users can create a new Vendor or Supplier inline during journal posting 
08 Rider transactions use the same party_type + party_id design (party_type = rider)
